# Overview: Hook Ingress Redesign

## Goal

Redesign the observation ingress boundary so SecondSight can ingest events from multiple coding agents without forcing each hook client to fabricate a full `HookEnvelope` using SecondSight-specific fields it cannot natively know.

The current failure is structural:

- coding-agent hooks emit valid agent-native JSON
- SecondSight currently requires `project_id`, `session_id`, `agent`, `event_id`, `timestamp`, `sequence_number`, `payload` at the HTTP boundary
- Claude Code hook scripts currently relay raw stdin JSON directly, so FastAPI rejects the request with `422`

This redesign makes the transport boundary match reality:

- hook scripts keep the raw agent payload intact
- hook source identifies which agent produced the event
- minimal ingress metadata is generated at the transport boundary
- adapters parse agent-specific raw payload and derive the canonical event fields SecondSight needs

## Design Decisions

### 1. Raw payload is source of truth

The raw hook payload from the coding agent is immutable source data. SecondSight must not destructively rewrite it before persistence.

SecondSight may enrich the request with transport metadata, but that metadata is outside the raw payload.

### 2. Agent identity comes from ingress source, not payload parsing

`agent` must be known before adapter dispatch. It cannot depend on parsing the payload shape.

For installed local hooks, the reliable source of truth is:

- which agent config tree the hook was installed under
- which wrapper script or endpoint is used
- optionally, explicit hook-script configuration

Payload-shape guessing is only a fallback heuristic and must not be part of the formal contract.

### 3. Adapters own semantic extraction

Adapters should parse agent-native payloads into SecondSight semantics:

- `session_id`
- `project_id`
- event-specific data

This keeps agent-specific parsing logic in one place and removes transport-specific parsing from shell scripts.

### 4. Ingress metadata is minimal and transport-owned

The transport layer is responsible only for metadata that the agent payload does not reliably provide:

- `event_id`
- `timestamp`
- `sequence_number`

`event_type` and `agent` come from the ingress source / route, not from payload parsing.

### 5. `sequence_number` is a per-session total-order counter

`sequence_number` is not `segment_index`.

In the current codebase it is the session-scoped total order key used for:

- DB uniqueness: `UNIQUE(session_id, sequence_number)`
- event ordering in repository reads
- stable replay and downstream analysis ordering

See:

- [src/secondsight/storage/events_table.py](/Users/yuyu_liao/vicone/SecondSight/src/secondsight/storage/events_table.py:27)
- [src/secondsight/storage/events_repository.py](/Users/yuyu_liao/vicone/SecondSight/src/secondsight/storage/events_repository.py:57)
- [src/secondsight/observation/tracker.py](/Users/yuyu_liao/vicone/SecondSight/src/secondsight/observation/tracker.py:247)

Because fallback replay can reorder delivery relative to real event occurrence time, `sequence_number` must be generated at the hook transport boundary, not at HTTP receive time.

## Proposed Ingress Contract

### Request shape

Replace the current "full HookEnvelope required at POST boundary" design with a thinner ingress contract:

```json
{
  "event_id": "deterministic-or-random-id",
  "timestamp": "2026-05-12T10:00:00Z",
  "sequence_number": 12,
  "payload": {
    "...": "raw agent-native JSON, unchanged"
  }
}
```

The missing routing metadata comes from the ingress path or equivalent source context:

- `agent`
- `event_type`

Recommended route shape:

- `POST /hook/{agent}/{event_type}`

This avoids payload guessing and removes the need to inject `agent` into the body only to turn around and use it for adapter dispatch.

## Field Derivation Rules

### `agent`

- Source: ingress source
- Rule: fixed by installed hook origin, route, or wrapper configuration
- Not parsed from raw payload

### `event_type`

- Source: ingress source
- Rule: fixed by which hook fired and which endpoint the hook posts to
- Adapter still cross-checks payload event markers such as `hook_event_name` to reject route/body mismatches

### `session_id`

- Source: raw payload
- Rule: adapter extracts from agent-native payload
- Example for Claude Code: raw `payload.session_id`

### `project_id`

- Source: raw payload
- Rule: adapter derives from `cwd`
- Default algorithm:
  - take `basename(cwd)`
  - slugify to `[A-Za-z0-9._-]`
- Recommended collision hardening:
  - `basename(cwd)` plus short stable hash of full `cwd`
- Requirement:
  - preserve explicit override path in the future for teams with repeated repo names

### `event_id`

- Source: ingress metadata
- Rule: generated by hook transport
- Recommendation:
  - deterministic ID preferred over random UUID
  - base on `agent + session_id_hint + sequence_number + event_type + payload_digest`
- Reason:
  - preserves idempotency under retry and fallback replay

### `timestamp`

- Source: ingress metadata
- Rule:
  - if agent provides a trustworthy native timestamp in future, transport may copy it
  - otherwise inject current UTC time when the hook fires

### `sequence_number`

- Source: ingress metadata
- Rule: transport-owned per-session monotonic counter
- Required behavior:
  - increment for every event in a session, across all event types
  - survive transient server outages and fallback replay
  - produce the same value whether the event is delivered immediately or replayed later

Recommended implementation:

- keep a small per-session state file under a local SecondSight state directory
- key the counter by `agent + session_id`
- update atomically before POST / fallback append

## Server and Adapter Responsibilities

### Hook transport

- read raw stdin JSON
- identify `agent` from install context
- identify `event_type` from invoked script
- generate `event_id`, `timestamp`, `sequence_number`
- POST ingress request
- on failure, append the same ingress record to fallback spool

### API layer

- validate ingress metadata shape
- resolve adapter using `(agent, event_type)` from route/source context
- pass raw payload plus ingress metadata into adapter
- keep path-safety validation for the adapter-derived `project_id` and `session_id`

### Adapter layer

- parse raw payload into:
  - `session_id`
  - `project_id`
  - canonical `EventType`
  - normalized event `data`
- cross-check route `event_type` against raw payload markers
- return `PartialEvent` with derived fields plus ingress metadata

### SessionTracker

No responsibility change:

- still owns `segment_index`
- still owns `sub_agent_id`
- still owns `depth`
- does not generate `sequence_number`

## Raw Durability Change

Current raw-trace storage writes normalized `Event` JSON, not the original ingress record. That means the current filesystem-first source of truth is actually the canonical event, not the agent-native raw payload.

This redesign should add a separate raw ingress durability layer:

- persist raw ingress record:
  - `agent`
  - `event_type`
  - `event_id`
  - `timestamp`
  - `sequence_number`
  - `payload`
- persist normalized canonical `Event` as today

This enables:

- adapter reprocessing after parser bugs
- schema evolution with replay
- forensic debugging of agent-specific ingestion bugs
